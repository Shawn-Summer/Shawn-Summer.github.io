这样配置好后,点击Rime中的sync user data选项后,会在sync_dir文件夹下创建一个installation_id文件夹,用来存储该用户的数据.

一般来说,在多设备情况下,sync_dir应当保持一致,例如选择云盘中的固定位置,而installation_id要保持不同,每个设备对应一个id. 这么做的意义是:每个设备的配置文件需要独立,而输入习惯需要共享.

在installation_id文件夹下,存放着

  1. 输入习惯:xxx.userdb.txt
  2. 配置文件:xxx.ymal和xxx.txt

配置文件的备份与迁移

配置目录下(一层)的所有yaml和txt文件就是配置文件:

  • default.yaml:一些全局设置

  • xxx.custom.yaml:补丁文件

  • xxx.schema.yaml:xxx输入方案的配置文件

  • xxx.dict.yaml:xxx输入方案的挂载词库

  • squirrel.yaml:鼠须管的前端配置文件

  • weasel.yaml:小狼毫的前端配置文件

  • symbols_v.yaml:全拼 v 模式配置文件

  • symbols_caps_v.yaml:双拼 V 模式配置文件

  • custom_phrase.txt:自定义短语文件

  • installation.yaml:同步设置文件

  • user.yaml:用户状态记录

以上文件在点击sync user data后都会直接拷贝到installation_id下,是一种单向备份,迁移时,其实只要将补丁文件迁移过去就行了,其他的直接使用GitHubgit clone就好了.

输入习惯的同步

输入习惯保存在配置目录下的 *.userdb/ 文件夹中,在点击sync user data后,installation_id文件夹下会生成一个*.userdb.txt文件.

*.userdb.txt文件是一个输入习惯的快照,它整合了sync_dir下所有用户的输入习惯,形成的一个快照.

具体来说,A用户点击sync user data后,rime会检查sync_dir下所有用户的对应同名快照文件,如rime_ice.userdb.txt文件,然后整合用户B和C的rime_ice.userdb.txt文件以及用户A的rime_ice.userdb/文件夹下内容,使用一种最高权重算法生成用户A自己的rime_ice.userdb.txt.

而且,这是一种双向同步,也就是说,如果我把配置目录下的xxx.userdb/文件夹删了,但是同步文件夹下的xxx.userdb.txt没有删,此时点击sync user data后,xxx.userdb/文件夹还会恢复.

所以说,迁移的时候,可以使用新id也可以使用存在的id,只要保证输入习惯快照存在,就可以恢复输入习惯.

小仓鼠输入法

小仓鼠（Hamster）是基于 RIME 的 iOS 上的输入法。

参考：https://ihsiao.com/apps/hamster/docs/

hamster 的简单框架

小仓鼠和鼠须管的区别在于：

  • 鼠须管的配置目录为 Rime，而小仓鼠中配置目录为：sharedSupport 和 Rime。因为小仓鼠把自己的客户端配置文件（即 hamster ）移动到 sharedSupport 中。
  • 小仓鼠的配置文件有两个副本，分别为应用文件和键盘文件。鼠须管只有键盘文件，即配置目录下就是键盘文件。在使用输入法的同时，键盘文件中的自造词或一些脚本文件会发生变化，而应用文件不会变化，所以应用文件对于小仓鼠来说相当于一种备份。
  • 小仓鼠中有键盘皮肤，而鼠须管只有键盘主题。小仓鼠中的键盘皮肤很复杂，而且它取代了配置文件中的很多功能，键盘皮肤已经玩出花了。

VSCode 代码编译和运行/调试逻辑

vscode 的工作逻辑是基于 tasks.json 和 launch.json 文件的。

编译逻辑（Build）→ tasks.json

  • VS Code 的「任务 (Task)」系统负责执行编译命令，比如调用 g++ 或 clang++。
  • 这些命令、参数、工作目录，以及错误匹配器，都写在 .vscode/tasks.json 里。
  • 调试配置中的 "preLaunchTask" 会触发这里定义的任务。

运行/调试逻辑 (Launch & Debug) → launch.json

  • VS Code 的调试系统遵循 Debug Adapter Protocol，需要知道要启动哪个可执行、带什么参数、在哪个目录、用哪个调试器。
  • 这些信息都在 .vscode/launch.json 里的 configurations 数组里定义。
  • 你按 F5、或左侧栏的 ▶️、或 CodeLens 的 “Debug C/C++ File” 都会读取这份配置来启动调试会话。

没有这tasks.json 和 launch.json 文件，VS Code 也能用「临时内置配置」来快速编译和调试单文件（尤其是 C/C++ 扩展加了 CodeLens）

上图为 python 插件添加的 CodeLens ，除了最后一个选项，都是使用「临时内置配置」来运行和调试单文件的，它们不会生成和利用 launch.json 文件。

一般情况下，我们都是使用F5、或左侧栏的 ▶️ 来运行和调试代码，对于快速的任务可以直接使用 CodeLens 来快速运行和调试。

tmux

tmux是cs架构，首先打开服务器，然后创建客户端，即会话，可以创建多个客户端。tmux服务器的父进程是1，即终端kill后，服务器仍然运行，自然客户端也在运行。

当终端通过ssh连接到服务器后，启动服务器上的tmux 服务器，然后在创建客户端，在客户端中运行程序，如果ssh断开，相当于这个客户端被detach了，但是由于服务器上tmux服务器还在运行，所以不会导致客户端中程序中断。

tmux，可以开启多个客户端（会话），一个会话可以开多个windows，一个window又可以开多个pane。一个pane相当于新建一个shell。

Tmux和iterm2集成(可选)

参考： https://blog.twofei.com/883/#integration

iterm2本身具有 创建new window 和 new tab的功能，相当于tmux的prefix + c 创建一个window。

iterm2具有的分屏功能，相当于tmux 的将window分成pane的功能

Tmux和iterm2的集成就是使用命令 tmux -CC后，会创建一个新的会话，然后在新的会话中禁用tmux的快捷键，而使用iterm2的自带的分屏创建窗口（tab）快捷键。这样能优化页面显示，使得iterm2更好适配tmux，并且使用更加丝滑。

使用iterm2集成后，每个pane都是一个iterm的模块，更好适配了。

Tmate (可选)

参考：https://www.cnblogs.com/idorax/p/12380758.html

Tmate做的是共享屏幕，共同编辑的功能，可以使用腾讯会议来代替，而且Tmate版本有点落后，对tmux的支持不是很好。

所以tmate唯一的亮点就是 共同编辑功能，这个功能挺鸡肋的，所以做快捷的演示可能比较好，即临时共享终端。

Tmate是Tmux的fork版本，它和tmux共享配置，它的服务器和客户端之间使用ssh连接

输入tmate后会显示如下信息 （代表4种连接方式）：

进入后可以将tmux的配置文件复制到 ~/.tmate.conf中才能使用。
